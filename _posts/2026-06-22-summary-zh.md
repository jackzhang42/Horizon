## [宁愿重复，也不要用错误的抽象](https://sandimetz.com/blog/2016/1/20/the-wrong-abstraction) ⭐️ 8.0/10

在 2016 年的一篇博文中，Sandi Metz 认为开发者应该容忍代码重复，而不是引入错误的抽象，因为错误的抽象会让代码难以修改。 这一原则挑战了盲目应用 DRY（不要重复自己）的做法，并警告过早引入抽象往往导致代码僵化，难以重构。它鼓励务实的态度，在稳定的抽象出现之前，重复是可以接受的。 Metz 强调，只有在清晰的模式出现时才应引入抽象，而错误的抽象代价远高于重复。文章建议在重复被证明有益之后，再向抽象重构。

hackernews · rafaepta · 6月21日 16:08 · [社区讨论](https://news.ycombinator.com/item?id=48620090)

**背景**: 这篇文章是软件工程中的经典之作，基于代码重复通常比过早抽象更安全的思想。它与‘你不需要它’（YAGNI）原则相联系，鼓励迭代设计。

**社区讨论**: 评论显示了不同观点：一些人同意重复比过度工程更容易管理，而另一些人提醒不要违反‘单一真相来源’原则。一位用户指出，函数式编程和 TypeScript 减少了抽象问题，但孤立的开发是导致重复的更大原因。

## [苹果通过 KernelKit 将 Swift 集成到 macOS 内核](https://blog.calif.io/p/apple-internals-swift-in-the-kernel) ⭐️ 8.0/10

苹果在 macOS 27 和 iOS 27 中引入了 KernelKit，使得内核组件可以用 Swift 编写。通过对 macOS 27 内核缓存的逆向工程发现，在 /System/KernelKit 目录下有一个 Embedded Swift 运行时被静态链接到 pthread 内核扩展中。 这标志着内核开发的重大转变，将 Swift 的内存安全和强类型等现代特性引入底层系统编程。它可能减少漏洞并提高 macOS 和 iOS 内核生态系统的开发效率。 目前，Swift 内核运行时仅存在于 macOS 且未被引用，这表明苹果正在测试基础设施，然后再部署基于 Swift 的内核组件。推出顺序似乎是：先发布 SDK 和运行时，然后在后续测试周期中引入 Swift 组件。

rss · Lobsters · 6月21日 08:41

**背景**: 内核是操作系统的核心，管理硬件和系统资源。传统上，内核用 C 语言编写以追求性能和控制力。Swift 是 Apple 开发的现代语言，以安全性和性能著称，但此前从未用于内核内部。Embedded Swift 是一种编译模式，允许 Swift 在无操作系统的裸机上运行。

## [SOCKMAP：基于内核的 TCP 拼接技术提升代理性能](https://blog.cloudflare.com/sockmap-tcp-splicing-of-the-future/) ⭐️ 8.0/10

Cloudflare 发布了一篇详细的技术博客，解释了 SOCKMAP 这一 Linux 内核特性，它利用 eBPF 在内核空间中完全执行 TCP 拼接，从而绕过用户态代理。 通过将 TCP 拼接移入内核，SOCKMAP 显著降低了代理服务器的延迟和 CPU 开销，这对于 CDN 和负载均衡器等高性能网络基础设施至关重要。 SOCKMAP 在 Linux 内核 4.14 中被引入，由 Cilium.io 的 John Fastabend 创建。它将 Strparser 接口暴露给 eBPF 程序，实现了套接字级别的重定向和策略执行。

rss · Lobsters · 6月21日 01:42

**背景**: TCP 拼接是一种在两个 TCP 连接之间中继数据而无需在内核与用户空间之间复制数据的技术，通常用于加速代理。传统的拼接需要内核修改或专用硬件。SOCKMAP 利用 eBPF（扩展的伯克利包过滤器）附加小型程序，在套接字级别重定向数据，从而实现接近路由器速度的性能。
